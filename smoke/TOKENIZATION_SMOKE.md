# Sarf Atlas Tokenization Smoke

This smoke workflow checks the narrow Sarf Atlas -> Ember -> `llama.cpp`
tokenization bridge. It is a local reproducibility check for plumbing and token
ID parity only.

## What It Proves

- Ember can load a Sarf Atlas extraction config and dispatch the
  `llama-cpp-external` backend.
- The external backend writes and passes a `--request <llama_cpp_request.json>`
  file to `sarf-atlas/smoke/llama_cpp_tokenize_extract.py`.
- The helper can call a local `llama-tokenize` binary against a local GGUF model.
- Ember receives artifact-contract-shaped sample, tokenization, position,
  manifest, report, and checksum files.
- `metadata.llamacpp.json` can be compared against tokenizer JSON metadata with
  `gguf-parity-tools token-audit`.

## What It Does Not Prove

- It does not generate text.
- It does not compute logits.
- It does not extract hidden states.
- It does not validate model behavior, paper results, datasets, plots, or
  numerical claims.
- It is not research output and must not be cited as evidence for a manuscript
  result.

## Required Local Inputs

Keep machine-local paths out of committed configs. Use environment variables,
a private `.local.toml`, or the temporary config generated by
`run_tokenization_parity_smoke.sh`.

Required files and tools:

- A local GGUF model file, supplied as `MODEL_PATH`.
- A matching tokenizer JSON file, supplied as `TOKENIZER_JSON`.
- A local `llama-tokenize` executable, supplied as `LLAMA_TOKENIZE_BIN`.
- A local checkout or install path for `gguf-parity-tools`, supplied as
  `GGUF_PARITY_TOOLS_PATH`.
- Ember built from this repository with `cargo run -- extract`.
- Python dependencies already available locally, including `tokenizers`.

The workflow must not download models or install dependencies.

## Ember External Helper Flow

The Ember config uses:

```toml
backend = "llama-cpp-external"
layers = []
write_logits = false
llama_cpp_binary = "scripts/llama_cpp_tokenize_extract.py"
```

`llama-cpp-external` currently rejects hidden-state extraction unless
`layers = []`. Ember writes `llama_cpp_request.json` in the run directory and
spawns the helper as:

```bash
scripts/llama_cpp_tokenize_extract.py --request /tmp/sarf-atlas-ember-smoke/runs/tokenization-parity-smoke/llama_cpp_request.json
```

The helper reads the request, renders prompts from
`sarf-atlas/smoke/prompts.small.jsonl`, and calls:

```bash
"${LLAMA_TOKENIZE_BIN}" --model "${MODEL_PATH}" --prompt "<rendered prompt>" --ids --log-disable
```

It writes Ember smoke artifacts plus `metadata.llamacpp.json` in the run
directory. The metadata is in the `gguf-parity-tools token-audit` shape and is
marked:

```json
{
  "real_llama_cpp": true,
  "real_tokenization": true,
  "no_generation": true,
  "no_logits": true,
  "no_hidden_states": true,
  "not_research_output": true
}
```

## Tokenizer JSON Metadata

Export comparable tokenizer JSON metadata with:

```bash
python3 sarf-atlas/smoke/export_tokenizer_json_metadata.py \
  --tokenizer /path/to/tokenizer.json \
  --prompts sarf-atlas/smoke/prompts.small.jsonl \
  --out /tmp/sarf-atlas-ember-smoke/runs/tokenizer-json/metadata.tokenizer-json.json
```

This uses the tokenizer JSON only. It does not call `llama.cpp`, generate text,
compute logits, or extract hidden states.

## Full Local Flow

The preferred repeatable local flow is the guarded runner:

```bash
MODEL_PATH=/path/to/model.gguf \
TOKENIZER_JSON=/path/to/tokenizer.json \
LLAMA_TOKENIZE_BIN=/path/to/llama-tokenize \
GGUF_PARITY_TOOLS_PATH=/path/to/gguf-parity-tools \
OUT_ROOT=/tmp/sarf-atlas-ember-smoke \
bash sarf-atlas/smoke/run_tokenization_parity_smoke.sh
```

Equivalent expanded commands:

```bash
cargo run -- extract \
  --config /tmp/sarf-atlas-ember-smoke/llama_cpp_tokenization_smoke.local.toml

cargo run -- validate-run \
  /tmp/sarf-atlas-ember-smoke/runs/tokenization-parity-smoke

python3 sarf-atlas/smoke/export_tokenizer_json_metadata.py \
  --tokenizer /path/to/tokenizer.json \
  --prompts sarf-atlas/smoke/prompts.small.jsonl \
  --out /tmp/sarf-atlas-ember-smoke/runs/tokenizer-json/metadata.tokenizer-json.json

PYTHONPATH=/path/to/gguf-parity-tools \
python3 -m parity_tools token-audit \
  --candidate-metadata /tmp/sarf-atlas-ember-smoke/runs/tokenizer-json/metadata.tokenizer-json.json \
  --reference-metadata /tmp/sarf-atlas-ember-smoke/runs/tokenization-parity-smoke/metadata.llamacpp.json \
  --out /tmp/sarf-atlas-ember-smoke/runs/tokenization-parity-smoke/token-audit.json

python3 -m json.tool \
  /tmp/sarf-atlas-ember-smoke/runs/tokenization-parity-smoke/token-audit.json
```

## Expected Pass Condition

The expected pass condition is:

1. `cargo run -- validate-run` exits successfully for the Ember artifact run.
2. `gguf-parity-tools token-audit` exits successfully and its report shows
   matching token IDs for every prompt in `prompts.small.jsonl`.

If the audit reports mismatches, treat the run as a tokenizer parity failure or
configuration mismatch. Do not reinterpret a mismatch as model behavior.

## Validation Layers

This smoke uses two of Ember's validation layers plus one external audit layer:

- `validate-run` checks this one Ember artifact run for structural honesty:
  manifest, report, tokenization rows, positions, backend identity, and
  provenance markers.
- `validate-backends` compares two Ember artifact runs. It is not required for
  this tokenization-only smoke unless a second comparable Ember run exists.
- `gguf-parity-tools` performs the external token-ID audit here. The same
  external harness is the intended place for logits and future layer parity
  checks.

## Known Limitations

- Only three small smoke prompts are covered.
- Offsets from `llama-tokenize` are not audited here.
- The helper assumes `llama-tokenize --ids` prints a Python-list-shaped token ID
  sequence.
- Architecture metadata is currently smoke metadata, not a broad model
  compatibility claim.
- Hidden-state extraction remains intentionally unimplemented for
  `llama-cpp-external`.

## Related Logits Smoke

The logits smoke is separate from this tokenization parity smoke. It uses
`scripts/llama_cpp_python_logits_extract.py`, which is a llama-cpp-python /
libllama binding path, not the standalone `llama-tokenize` binary path.

Run it only as a tiny local plumbing check:

```bash
MODEL_PATH=/path/to/model.gguf \
OUT_ROOT=/tmp/sarf-atlas-ember-smoke \
bash sarf-atlas/smoke/run_logits_python_smoke.sh
```

It requests `write_logits = true`, writes `logits.npy`, runs `validate-run`, and
marks `real_logits = true`, `no_generation = true`, `no_hidden_states = true`,
and `not_research_output = true`. It does not establish logits parity unless a
separate reference logits artifact is available for `gguf-parity-tools
compare-logits`.

The native reference runner for that comparison is:

```bash
MODEL_PATH=/path/to/model.gguf \
TOKENIZER_JSON=/path/to/tokenizer.json \
OUT_ROOT=/tmp/sarf-atlas-ember-smoke \
bash sarf-atlas/smoke/run_native_logits_reference_smoke.sh
```
